## 2. 第一阶段：神经网络鲁棒性基础

这一阶段的目标不是深挖所有证明细节，而是先建立项目中最常用的概念框架。

### 2.1 必须掌握的核心概念

#### 2.1.1 分类网络与分类 margin

对一个分类网络而言，输入是图像张量，输出是每个类别的 logit。预测类别通常取最大 logit 对应的类别。

在鲁棒性验证中，更重要的不是“预测对不对”，而是“正确类别 logit 是否始终大于其他类别 logit”。因此常见验证目标会写成：

对于所有允许扰动后的输入 x'，都有

`f_y(x') - f_j(x') > 0, 对任意 j != y`

这说明在整个扰动集合内，正确类别始终占优，样本就被证明为鲁棒。

#### 2.1.2 扰动模型

项目中最常见的是 L∞ 扰动：

`||x' - x||∞ <= epsilon`

它的含义是每个像素都只能在原值附近上下浮动 epsilon。

为什么课程项目优先用 L∞：

1. 图像分类鲁棒性验证中最常用。
2. alpha-beta-CROWN 对该设定支持成熟。
3. 与 MNIST、CIFAR-10 的已有 benchmark 最容易对齐。

除了 L∞，你还需要知道：

1. L2：约束总能量，更平滑，但验证往往更难。
2. L0：限制最多修改多少个像素，更适合稀疏攻击分析。

#### 2.1.3 局部鲁棒性与全局鲁棒性

课程项目里做的主要是局部鲁棒性：

对单个样本 x，在半径 epsilon 的扰动球内，网络输出类别是否保持不变。

统计很多样本后，就能得到 Verified Robust Accuracy，简称 VRA：

`VRA = 被证明鲁棒的样本数 / 总样本数`

VRA 是你们报告中最核心的指标之一。

#### 2.1.4 实证鲁棒性与验证鲁棒性

这两个概念必须严格区分：

1. 实证鲁棒性：用 PGD、CW 等攻击没找到反例。
2. 验证鲁棒性：用形式化方法证明所有允许扰动都满足性质。

没有攻击找到反例，不代表一定鲁棒。
形式化证明鲁棒，才是真正的“证书”。

### 2.2 为什么 ReLU 网络适合做形式化验证

ReLU 定义为：

`ReLU(z) = max(0, z)`

它是分段线性的：

1. 当 z >= 0 时，函数是线性的，斜率为 1。
2. 当 z < 0 时，函数是线性的，斜率为 0。

因此整个 ReLU 网络虽然整体是非线性的，但可以看成由很多线性区域拼接而成。

这带来两个重要后果：

1. 可以用线性松弛近似它。
2. 可以通过枚举或分支确定每个 ReLU 的开关状态，从而把问题转化为线性约束求解。

这就是后面 alpha-CROWN、beta-CROWN、BaB、MIP、Marabou 等方法能工作的根本原因。

### 2.3 这一阶段学完后，你应当能回答的问题

1. 为什么项目里默认使用 L∞ 扰动。
2. 为什么 VRA 不是普通分类准确率。
3. 为什么“攻击没找到反例”不等于“已证明鲁棒”。
4. 为什么 ReLU 比一般非线性激活更容易做验证。
5. 为什么同一个样本在不同 epsilon 下验证结果会变化。

### 2.4 建议练习

1. 自己写出局部鲁棒性、VRA、对抗样本三个概念的定义。
2. 用一张 2 维平面示意图理解 L∞ 球和 L2 球的区别。
3. 画出单个 ReLU 的图像，并说明为什么它是分段线性的。

## 3. 第二阶段：形式化验证基础

这一阶段的目标是理解“验证器到底在证明什么，以及为什么会有 complete / incomplete 的区别”。

### 3.1 形式化验证的基本目标

验证器接收三个要素：

1. 神经网络模型。
2. 输入集合。
3. 输出性质。

然后回答：

1. 性质是否对输入集合中的所有点都成立。
2. 如果不成立，能否给出反例。
3. 如果算不完，是否返回 unknown 或 timeout。

因此验证本质上是一个“全称量化”的问题：

`对于所有允许输入，性质是否成立`

这比普通前向推理难得多。

### 3.2 Sound、Complete、Incomplete

这三个词必须吃透。

#### 3.2.1 Sound

Sound 意味着结论可靠。

如果验证器说“robust”，那就必须真的 robust。
如果验证器给出反例，那反例必须真的能破坏性质。

#### 3.2.2 Complete

Complete 意味着算法理论上能穷尽问题空间。

对 complete verifier 来说，在资源足够时，最终会得到：

1. robust。
2. unsafe。

而不是永远停留在“我不知道”。

#### 3.2.3 Incomplete

Incomplete 方法通常只做保守近似。

它们的优点是快、可扩展。
缺点是遇到难例时，可能只能说“无法证明”，但这并不代表样本一定不鲁棒。

### 3.3 常见验证范式

#### 3.3.1 抽象解释 / 线性松弛

代表方法：IBP、CROWN、DeepPoly、alpha-CROWN。

核心思想：

1. 不直接枚举所有输入。
2. 而是传播每层激活的上下界。
3. 用线性上下界逼近 ReLU 等非线性。

优点：

1. 快。
2. 适合大网络。
3. GPU 加速友好。

缺点：

1. 可能过松。
2. bounds 不够紧时，无法证明样本鲁棒。

#### 3.3.2 Branch and Bound

代表方法：beta-CROWN、BaB、一些分支定界式完整验证器。

核心思想：

1. 先算全局松弛下界。
2. 如果还不能判定，就把问题切成更小子域。
3. 对子域继续做 bound propagation。
4. 重复直到证明安全、找到反例、或耗尽时间。

这是课程项目里最重要的完整验证框架。

#### 3.3.3 MIP

混合整数规划通过整数变量显式表示 ReLU 的开关状态。

优点：

1. 小模型时非常精确。
2. 能作为强基线或精确对照。

缺点：

1. 随网络规模增大，求解成本暴涨。
2. CIFAR-10 级别的高维网络通常很难直接大规模跑。

#### 3.3.4 SMT / 可满足性求解

Marabou 一类方法常把 ReLU 网络编码成约束系统，再结合分裂和约束传播求解。

优点：

1. 理论上完整。
2. 对小型 ReLU MLP 常常很有解释力。

缺点：

1. 大规模图像模型上通常不如 GPU 友好的 bound propagation 路线可扩展。

### 3.4 为什么验证会难

根本原因是 ReLU 的激活模式数目随网络规模指数增长。

如果一个网络有很多不稳定 ReLU，即某个神经元在允许扰动下可能激活也可能关闭，那么验证器就要面对大量可能的线性区域。

验证困难通常来自以下因素：

1. 网络更深更宽。
2. 不稳定 ReLU 更多。
3. epsilon 更大。
4. 输入维度更高。
5. 分类 margin 更小。

这也是为什么 CIFAR-10 往往比 MNIST 更难验证。

### 3.5 这一阶段学完后，你应当能回答的问题

1. 什么叫 sound，什么叫 complete。
2. 为什么不完整方法很快，但不能替代完整验证。
3. 为什么 BaB 可以把不完整方法“升级”为完整验证的一部分。
4. 为什么 MIP 和 SMT 更适合小模型强对照。
5. 为什么 epsilon 增大后，timeout 往往会上升。

### 3.6 建议练习

1. 用自己的话区分“无法证明”和“已找到反例”。
2. 解释为什么松弛越紧，验证通常越容易成功。
3. 解释为什么分支启发式会影响运行时间和节点数。

## 4. 第三阶段：核心方法对比知识

这一阶段进入项目最关键的算法层比较。

## 4.1 alpha-CROWN

alpha-CROWN 属于优化型线性松弛 bound propagation 方法。

你可以把它理解为：

1. CROWN 给出一种线性上界和下界传播方式。
2. alpha-CROWN 进一步优化松弛参数，让边界更紧。

它的定位：

1. 通常作为不完整验证主力。
2. 也常用于完整验证前的预处理，给后续 BaB 提供更好的初始 bounds。

优点：

1. GPU 友好。
2. 对大网络比传统精确方法更可扩展。
3. 能显著提升 bound 质量。

局限：

1. 如果边界还是不够紧，就只能返回“无法直接证明”。

## 4.2 beta-CROWN

beta-CROWN 可以理解为“在 alpha-CROWN 基础上，引入 split constraints 的完整验证路线”。

它的核心思想是：

1. 对不稳定 ReLU 引入分支。
2. 在每个分支子域中继续优化 bounds。
3. 用更紧的边界配合 Branch and Bound 做完整验证。

它是 alpha-beta-CROWN 项目名中的“beta”部分，也是课程项目主实验最应关注的对象。

## 4.3 Branch and Bound

BaB 不是单独一个 bound 公式，而是一种搜索框架。

它的关键组成包括：

1. 当前子域的上下界计算。
2. 选择哪个子域先扩展。
3. 选择哪个 ReLU 先分支。
4. 何时剪枝。

因此，BaB 的性能不仅受 bound 质量影响，也受启发式策略影响。这正是你们后续可以做创新的地方。

## 4.4 bab-refine

bab-refine 的思想是：

1. 先用更强但更贵的方法收紧中间层 bounds。
2. 再进入 BaB。

它常用于小到中等规模、且需要更强中间界的网络。

对于课程项目，bab-refine 很适合作为：

1. 与普通 `bab` 的强度对照。
2. 与 `mip` 的中间位置策略。

## 4.5 MIP

MIP 适合做“小模型精确裁判”。

在课程报告中可以把它作为：

1. 强精确基线。
2. 用于验证某些样本到底是真的 unsafe 还是只是 bound 太松。
3. 说明“为什么完整方法更贵但更有说服力”的证据。

## 4.6 Marabou

Marabou 是神经网络验证经典工具之一，对 ReLU 网络非常有代表性。

它在课程项目中的合理定位不是覆盖所有正式实验，而是：

1. 作为外部工具参照。
2. 在小型 MNIST FCNN 上做跨工具比较。
3. 展示不同验证范式的差异。

你需要掌握的不是 Marabou 的全部内部实现，而是：

1. 它属于完整验证工具。
2. 它的求解风格更接近约束求解。
3. 它与 alpha-beta-CROWN 的主要差异在于可扩展性和加速路径。

## 4.7 方法对比时应该比较什么

不要只比较“谁能跑完”。

更合理的比较维度包括：

1. VRA。
2. 平均时间。
3. 中位时间。
4. timeout 率。
5. 搜索节点数。
6. 能处理的模型规模。
7. 对不同 epsilon 的稳定性。

## 4.8 这一阶段学完后，你应当能回答的问题

1. alpha-CROWN 为什么快。
2. beta-CROWN 为什么更接近完整验证。
3. BaB 的瓶颈为什么常在分支数和 bounds 质量。
4. MIP 和 Marabou 为什么适合做小模型对照。
5. 为什么“更完整”通常意味着“更慢”。

## 6. 第五阶段：实验设计知识

学会工具还不够，你还必须知道如何设计一个“可比”的实验。

### 6.1 实验设计的基本原则

#### 6.1.1 控制变量

比较两种验证策略时，至少要固定：

1. 模型。
2. 数据集。
3. 样本编号。
4. epsilon。
5. timeout。
6. 硬件环境。

否则时间和 VRA 差异没有解释意义。

#### 6.1.2 分层实验

建议分三层：

1. 烟雾测试：20 个样本，检查流程正确性。
2. 小规模试跑：100 个样本，筛选配置与策略。
3. 正式统计：数百个样本，形成报告主结论。

#### 6.1.3 先易后难

建议始终遵循：

1. 先 MNIST。
2. 再 MNIST 自训练 FCNN。
3. 再 CIFAR-10 小型 FCNN。
4. 最后再考虑跨工具和创新扩展。

### 6.2 你必须会解释的指标

#### 6.2.1 VRA

证明鲁棒的比例。

它是最重要的安全性指标，但不能单独看，因为：

1. 某些方法只是更保守。
2. 某些方法只是 timeout 太多。

#### 6.2.2 平均时间与中位时间

平均时间反映总体成本，但容易被少数极难样本拉高。
中位时间更能反映“典型样本”的求解成本。

#### 6.2.3 timeout 率

对完整验证来说，timeout 是必须重点分析的指标。

因为 timeout 太高时，即使理论上 complete，实际工程价值也会下降。

#### 6.2.4 节点数

节点数能更直接反映 BaB 搜索成本。

如果某个启发式节点数显著下降，通常说明它更有效率。

### 6.3 为什么要先做攻击再做验证

攻击不是用来替代验证，而是用来快速筛掉明显不鲁棒样本。

这样做的好处：

1. 节省完整验证资源。
2. 更快找到 unsafe 样本。
3. 提高实验吞吐量。

但在报告里必须写清楚：

1. 攻击命中反例时，是“经验发现 unsafe”。
2. 形式化证明才是最终证据。

### 6.4 这一阶段学完后，你应当能回答的问题

1. 为什么不能用不同样本集比较两种策略。
2. 为什么要同时看 VRA 和 timeout。
3. 为什么正式实验前要先做 20 样本烟雾测试。
4. 为什么 CIFAR-10 往往需要更严格控制计算预算。

## 7. 第六阶段：改进与研究表达知识

课程项目不仅要会跑，还要会讲“为什么这样改”和“改得是否有效”。

### 7.1 适合当前课题的改进方向

#### 7.1.1 深度感知分支启发式

思路：

1. 在分支评分中引入层深信息。
2. 对更靠后层的不稳定 ReLU 给予额外权重。

适合原因：

1. 当前仓库已有 babsr 和 kfsb 实现。
2. 你们只需要在已有评分基础上增量修改。
3. 容易做消融和回退。

#### 7.1.2 自适应双评分策略

思路：

1. 先用低成本评分做粗筛。
2. 再用高质量评分对 top-k 候选精排。

适合原因：

1. 能平衡评分质量与额外开销。
2. 更容易解释“为什么更快”。

#### 7.1.3 两阶段验证协议

思路：

1. 先用不完整方法快速证明一批样本。
2. 只把难例交给完整方法。

它的价值更偏工程流程优化，但在课程项目中往往非常实用。

### 7.2 如何证明改进有效

至少要做下面三件事：

1. baseline 与改进版在同一实验协议下对比。
2. 给出不止一个 epsilon 的结果。
3. 分析成功样本和失败样本，而不是只报平均值。

### 7.3 如何写出像研究而不是流水账的分析

分析结论最好遵循这个模板：

1. 观察到了什么现象。
2. 这种现象在什么条件下成立。
3. 可能的原因是什么。
4. 它对后续实验或部署意味着什么。

示例：

“在 MNIST 0-100、epsilon=0.02 的设置下，kfsb 的平均时间低于 babsr，且 timeout 更少。说明在当前小型 FCNN 上，kfsb 的分支优先级选择更能减少无效搜索。但该结论是否能迁移到更高 epsilon 或 CIFAR-10，还需要进一步验证。”

### 7.4 这一阶段学完后，你应当能回答的问题

1. 什么样的改进是可验证、可比较、可汇报的。
2. 为什么改进必须做消融。
3. 为什么单次跑得更快不等于方法更好。
4. 为什么失败案例分析很重要。

## 9. 必备术语速查表

### 9.1 核心术语

1. 鲁棒性：扰动下预测保持不变的性质。
2. 局部鲁棒性：针对单个样本的鲁棒性。
3. Verified Robust Accuracy：被证明鲁棒的样本比例。
4. 不完整验证：能快速证明一部分样本，但不能穷尽所有情况。
5. 完整验证：理论上能穷尽搜索空间并给出最终结论。
6. 上下界传播：对网络中间层输出范围做安全估计。
7. 分支启发式：决定先对哪个不稳定 ReLU 分支。
8. timeout：在时间预算内未完成求解。

### 9.2 常见误区

1. “没有攻击成功”不等于“已证明鲁棒”。
2. “complete verifier”不等于“永远跑得完”。
3. “VRA 更高”不一定说明方法绝对更好，还要看时间成本与 timeout。
4. “平均时间更低”不一定说明方法更稳，还要看长尾难例。
5. “MNIST 上有效”不一定能迁移到 CIFAR-10。

## 12. 总结

这类课题最容易出问题的地方，不是代码本身，而是知识顺序学反了。

最常见的错误路线是：

1. 一上来就改启发式代码。
2. 没搞清楚 complete 与 incomplete。
3. 没统一实验协议就开始跑表。
4. 最后结果很多，但无法解释。

更合理的路线是：

1. 先建立鲁棒性与验证的统一概念。
2. 再弄懂 alpha-beta-CROWN 与 Marabou 的方法定位。
3. 再进入仓库入口、配置和日志。
4. 再做 MNIST 基线和策略对比。
5. 最后再谈改进、扩展和跨工具比较。

只要按这个顺序推进，你们的项目就会同时具备：

1. 理论解释力。
2. 工程可执行性。
3. 结果可复现性。
4. 汇报可讲述性。

这四点，正是课程项目最需要的能力组合。
