# 针对 ReLU 激活函数的全连接神经网络鲁棒性验证项目详细规划

## 0. 文档摘要

本规划书将课程题目拆解为可执行的实验主线、改进主线和质量保障主线，目标是在当前仓库环境内完成一份可落地、可复现、可答辩的课程项目。

本项目采用 alpha-beta-CROWN 作为主验证平台，先完成同一工具内部的多策略对比，再根据时间与环境条件决定是否补充 Marabou 外部对照。这样安排的原因是：当前仓库已具备验证主程序、MNIST MLP 配置、启发式入口和 API 文档，但没有内置 Marabou，也没有现成的 CIFAR-10 全连接模型权重。

最终交付物将包括：实验协议、验证配置、批处理脚本、原始日志、统计图表、改进实现、汇报材料与最终报告。

## 1. 项目任务理解

### 1.1 原始要求拆解

1. 研究对象是现有分类网络，激活函数为 ReLU。
2. 主体网络应聚焦全连接神经网络，而不是卷积网络主实验。
3. 需要参考 Marabou 或 alpha-beta-CROWN 等验证工具。
4. 核心任务是比较不同验证策略，而不是只运行单一配置。
5. 需要在 MNIST、CIFAR-10 等数据集上研究不同扰动半径下的鲁棒性。
6. 鼓励提出更优的验证思想，并用实验展示改进效果。

### 1.2 项目执行结论

1. 主平台选用 alpha-beta-CROWN。
2. 强制完成的比较对象是不同验证策略，最低要求不依赖外部工具。
3. Marabou 作为增强项，不作为主线阻塞条件。
4. 数据集按两阶段推进：先完成 MNIST，再扩展到 CIFAR-10。
5. 改进方向优先选择分支启发式，因为当前仓库已有明确代码入口。

### 1.3 成功定义

本项目视为完成，至少需要满足以下条件：

1. 在 MNIST 和 CIFAR-10 上各完成至少 1 个 ReLU 全连接分类网络的鲁棒性验证实验。
2. 至少比较 3 类策略或配置，并给出统一指标下的结果。
3. 输出 VRA 随 epsilon 变化的曲线，以及时间和超时率对比。
4. 至少实现 1 个改进策略，并完成消融或对照实验。
5. 全部结论可追溯到配置、日志、图表和代码版本。

## 2. 范围与非范围

### 2.1 范围内

1. ReLU 激活的全连接分类网络。
2. 以 L_inf 扰动为主的鲁棒性验证。
3. alpha-CROWN、beta-CROWN、bab-refine、MIP 等策略对比。
4. MNIST 与 CIFAR-10 的代表性样本子集实验。
5. 分支启发式改进与消融分析。
6. 满足课程汇报和质量保障要求的全流程证据整理。

### 2.2 不纳入本阶段主实验

1. CNN、ResNet、Transformer 作为主研究对象。
2. 非 ReLU 激活函数的系统性研究。
3. 大规模 VNN-COMP 全量复现。
4. 分布式性能优化。
5. 纯理论型创新而无实验支撑的方案。

## 4. 技术路线

### 4.1 主平台与调用方式

1. 主平台：alpha-beta-CROWN。
2. 主调用方式：YAML 配置加命令行，即 python abcrown.py --config 配置文件。
3. 辅助调用方式：Python API，用于单样本演示和后续补充实验。

### 4.2 最低可交付策略对比

建议至少比较以下 3 类策略：

1. 不完整验证：alpha-CROWN 或 safe-incomplete 路线。
2. 完整验证：beta-CROWN 加 Branch and Bound。
3. 小模型精确对照：bab-refine 或 MIP。

增强项：

4. 外部工具对照：Marabou。

### 4.3 启发式层面的比较对象

在完整验证内部，建议至少比较以下 3 种分支启发式：

1. babsr。
2. kfsb。
3. kfsb-intercept-only 或 intercept。

改进版本：

4. depth-aware-babsr，即深度感知分支版本。

### 4.4 统一输入输出协议

统一输入字段：

1. 模型名称。
2. 数据集名称。
3. 样本编号列表。
4. epsilon 列表。
5. 超时阈值。
6. 策略名称。
7. 启发式名称。
8. 随机种子。

统一输出字段：

1. sample_id。
2. true_label。
3. predicted_label。
4. epsilon。

7. result_status，取值为 verified、violated、unknown、timeout。

## 5. 数据集与模型规划

### 5.1 MNIST 规划

按由易到难的顺序推进：

1. 快速基线：使用 complete_verifier/exp_configs/beta_crown/mnist_tiny_mlp.yaml。
2. 自有标准 FCNN：使用 train_mnist_fcnn.py 训练并导出 saved_models/mnist_fcnn.pth 和 saved_models/mnist_fcnn.onnx。
3. 扩展基线：使用 mnist_6_100、mnist_9_100、mnist_9_200 等现成 FCNN 配置。

### 5.2 CIFAR-10 规划

当前仓库没有现成 CIFAR-10 全连接模型，因此采用如下方案：

1. 自行训练一个小型 ReLU 全连接 CIFAR-10 分类网络。
2. 参考 complete_verifier/custom/custom_model_data.py 中的 cifar10() 数据加载方式。
3. 训练后导出 PyTorch 权重，必要时导出 ONNX 用于跨工具对照。
4. 若 CIFAR-10 验证成本过高，则采用固定子样本集并在报告中说明原因。

### 5.3 模型选择原则

1. 激活函数必须为 ReLU。
2. 主体结构必须为 FCNN。
3. 必须能在课程周期内完成训练、导出和验证。
4. 必须能在同一模型上重复运行多种验证策略。

## 6. 实验设计

### 6.1 指标设计

1. Verified Robust Accuracy，简称 VRA。
2. Counterexample Rate，即找到反例的比例。
3. 平均求解时间。
4. 中位数求解时间。
5. p90 求解时间。
6. 超时率。
7. 搜索节点数。

### 6.2 epsilon 网格建议

MNIST：

1. 0.01。
2. 0.03。
3. 0.05。
4. 0.10。
5. 0.20。
6. 0.30。

CIFAR-10：

1. 1/255。
2. 2/255。
3. 4/255。
4. 8/255。

### 6.3 样本规模建议

1. 烟雾测试集：20 个样本，用于检查流程正确性。
2. 小规模对比集：100 个样本，用于参数和策略预筛选。
3. 正式统计集：MNIST 建议 500 个样本，CIFAR-10 建议 200 到 500 个样本。

### 6.4 最低实验矩阵

| 维度 | 最低要求 |
| --- | --- |
| 数据集 | MNIST、CIFAR-10 |
| 模型 | 每个数据集至少 1 个 ReLU FCNN |
| 验证策略 | 至少 3 类 |
| 启发式 | 至少 3 种 |
| epsilon | 每个数据集至少 4 个半径 |
| 样本数 | 每个数据集至少 100 个正式样本 |

### 6.5 推荐实验矩阵

| 数据集 | 模型 | 策略 | 启发式 | 备注 |
| --- | --- | --- | --- | --- |
| MNIST | tiny MLP | incomplete | 无 | 快速基线 |
| MNIST | tiny MLP | beta-CROWN | babsr、kfsb、intercept | 主对比 |
| MNIST | tiny MLP | MIP 或 bab-refine | 默认 | 精确对照 |
| MNIST | 自训 FCNN | beta-CROWN | baseline vs 改进 | 创新实验 |
| CIFAR-10 | 自训 FCNN | incomplete | 无 | 流程跑通 |
| CIFAR-10 | 自训 FCNN | beta-CROWN | baseline vs 改进 | 主结果 |

## 7. 改进方案设计

### 7.1 首选方向：深度感知分支启发式

核心思想：

1. 在原始 BaBSR 评分基础上，对更深层的不稳定 ReLU 节点赋予更高权重。
2. 直觉是更靠近输出层的分裂更可能直接影响最终分类边界。
3. 目标是在相同验证预算下减少无效分支和平均搜索节点数。

代码落点：

1. 分发入口在 complete_verifier/heuristics/branching_heuristics.py。
2. BaBSR 主实现位于 complete_verifier/heuristics/babsr.py。
3. KFSB 位于 complete_verifier/heuristics/kfsb.py。
4. 参数入口位于 complete_verifier/arguments.py。

工程判断：

1. 可行，因为现有分支方法已经被统一封装为可切换模式。
2. 风险较低，因为可以用开关参数保留 baseline 行为。
3. 适合作为课程项目中的“可验证改进”。

### 7.2 备选方向：主评分与备评分自适应切换

1. 当主评分信息不足时，提高 intercept backup score 的影响力。
2. 目标是减少随机分支比例，提升困难样本上的稳定性。
3. 该方向适合作为第二阶段增强，而不是一开始并行大改。

### 7.3 消融实验要求

至少完成以下 3 组：

1. baseline：原始启发式。
2. variant-A：仅引入深度权重。
3. variant-B：深度权重加自适应备评分。

每组至少输出：

1. VRA。
2. 平均时间。
3. 节点数。
4. 超时率。

## 8. 软件质量保障方案

### 8.1 质量目标

1. 正确性：同一配置重复运行时，结果状态和主要统计指标一致。
2. 可复现性：新机器可按文档重新搭建环境并得到同类结果。
3. 可追踪性：任意图表都能反查到配置、日志和代码版本。
4. 可审查性：老师或组员可依据归档材料复核关键结论。

### 8.2 质量门禁

Gate-1 环境门禁：

1. 环境安装完成。
2. abcrown.py 可执行。
3. 至少 1 个示例配置可正常跑通。

Done Criteria：

1. 环境版本记录齐全。
2. 留存首次成功运行日志。

Gate-2 流程门禁：

1. 单样本验证链路跑通。
2. 结果状态可区分 verified、violated、unknown。

Done Criteria：

1. 20 个样本烟雾测试通过。
2. 无阻塞性崩溃。

Gate-3 对比门禁：

1. 同一模型、样本集和 epsilon 下，至少两种策略可比较。
2. 指标字段统一。

Done Criteria：

1. 形成首版对比 CSV。
2. 至少输出 1 张图。

Gate-4 改进门禁：

1. 改进策略可开关。
2. baseline 与 variant 可在统一协议下运行。

Done Criteria：

1. 至少完成一轮消融。
2. 图表可复算。

Gate-5 归档门禁：

1. 配置、脚本、日志、图表、报告齐全。
2. 任一关键结论均能追溯到原始证据。

### 8.3 测试策略

1. 单元测试：日志解析、指标计算、参数读取。
2. 集成测试：单样本验证到结果汇总全链路。
3. 回归测试：改进前后在固定样本集上比较结果偏移。
4. 异常测试：超时、空文件、NaN、异常退出。

| CIFAR-10 FCNN 精度过低 | 中 | 高 | 模型准确率过低导致结论失真 | 缩小模型难度，先保证可用分类精度 | C |
| CIFAR-10 验证超时严重 | 高 | 高 | 超时率超过 40% | 缩小样本规模、减少 epsilon 密度 | D |
| Marabou 安装失败 | 中 | 中 | 多次尝试仍不可运行 | 不阻塞主线，降级为内部多策略对比 | A |
| 改进收益不显著 | 中 | 中 | 消融没有明显改善 | 转为负结果分析与失败模式讨论 | E |
| 配置口径不统一 | 中 | 高 | 不同成员产出字段不一致 | 冻结统一协议和配置模板 | A、C |
| 结果不可追溯 | 低 | 高 | 图表无法定位原始来源 | 强制保留 commit_id、配置和日志索引 | A |

## 13. 证据映射

| 结论 | 需要的证据 |
| --- | --- |
| 某策略验证更强 | 同一模型同一样本集下的 VRA 对比表 |
| 某策略更快 | 统一超时阈值下的运行时间统计图 |

## 17. 直接执行建议

如果从明天开始执行，建议顺序如下：

1. 先跑通 mnist_tiny_mlp，形成第一个成功基线。
2. 用 train_mnist_fcnn.py 训练并导出自有 MNIST FCNN。
3. 冻结统一日志格式和样本子集。
4. 跑 MNIST 上的 incomplete、beta-CROWN、MIP 三类策略。
5. 在 babsr.py 与 kfsb.py 的入口处实现最小改动版本。
6. 补 CIFAR-10 FCNN 训练与接入。
7. 完成双数据集主结果、消融和最终报告。

## 18. 结论

本项目最合理的推进方式不是一开始追求工具最全，而是先用当前仓库中已经具备的 alpha-beta-CROWN 能力做出一条稳固主线：MNIST FCNN 跑通、多策略可比、改进点可落地、证据链可复查。随后再把经验迁移到 CIFAR-10，并在时间允许时补充 Marabou 对照。这样既符合课程项目的质量要求，也更有利于在汇报中同时展示工具解析能力、实验复现能力和方法改进能力。
