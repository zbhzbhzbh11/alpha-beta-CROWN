## 3. 当前风险与规避

### 风险 R1：版本偏离官方推荐

- 现状：官方文档推荐 Python 3.11 + PyTorch 2.8，当前为 Python 3.10 + torch 2.4.1
- 影响：大规模实验时可能出现版本行为差异
- 规避：
  - 固定当前可运行环境作为课程基线
  - 每次实验记录环境指纹（Python/torch/CUDA）
  - 若后续出现兼容问题，再建立“对齐官方版本”的并行环境

### 风险 R2：从仓库根目录导入 API 时的路径冲突

- 现象：从仓库根目录直接导入 complete_verifier.api，可能触发 auto_LiRPA 包解析冲突
- 影响：API 方式运行不稳定
- 规避：
  - 命令行统一在 complete_verifier 目录下执行
  - 项目阶段1默认采用 CLI 路线

## 4. 项目结构与关键模块分析

### 4.1 执行入口层

- complete_verifier/abcrown.py
  - 角色：CLI 主入口
  - 作用：读取 YAML 配置，调度 attack/incomplete/complete/mip 等流程

- complete_verifier/api.py
  - 角色：新 Python API 入口
  - 作用：提供 ABCrownSolver、VerificationSpec、ConfigBuilder 等编程接口

### 4.2 核心求解层

- complete_verifier/beta_CROWN_solver.py
  - 角色：完整验证主求解器接口（含 BaB 相关流程）

- complete_verifier/incomplete_verifier_func.py
  - 角色：不完整验证流程（如 alpha-CROWN 路线）

- complete_verifier/complete_verifier_func.py
  - 角色：完整验证流程封装

### 4.3 策略与启发式层

- complete_verifier/heuristics/branching_heuristics.py
  - 角色：分支策略入口分发

- complete_verifier/heuristics/babsr.py
  - 角色：BaBSR 评分与决策实现

- complete_verifier/heuristics/kfsb.py
  - 角色：KFSB 分支策略实现

### 4.4 配置与实验资产层

- complete_verifier/exp_configs/
  - 角色：实验配置库

- complete_verifier/mnist_baseline.yaml
  - 角色：课程阶段最小可复现基线配置

### 4.5 端到端执行流程（PPT 可直接使用）

流程主线：

1. CLI/API 读取配置并初始化随机种子与设备
2. 加载 ONNX/数据/规格，构建 vnnlibHandler
3. 先做 incomplete verification（alpha-CROWN）
4. 若未直接证明，再进入 complete verification（BaB）
5. 汇总每个样本的 safe/unsafe/unknown 与耗时

关键函数链路：

- abcrown.py::ABCROWN.main
- incomplete_verifier_func.py::incomplete_verifier
- complete_verifier_func.py::complete_verifier
- complete_verifier_func.py::bab
- bab.py::general_bab

### 任务 T5：具体改进点候选（供选择，不再只是复现）

老师指出“改进点要细化”，所以后续不能再只写“继续做复现”。结合当前代码结构和已有实验结果，建议从下面三类里选 1 个作为正式改进方向：

#### 方案 A：分支策略优化（最推荐）

核心想法：不改验证框架本身，而是围绕 branching 配置做更细粒度的策略优化，让改进落在“怎么分支、何时分支、分多少候选”上。

可操作点：

1. 固定模型、样本区间和 epsilon，调节 `branching.method`、`branching.candidates`、`branching.reduceop`。
2. 对比 `babsr`、`kfsb`、`auto` 的结果，观察谁在 timeout、mean time 和 visited nodes 上更优。
3. 进一步测试 `input_split.enable` 与 `input_dim_threshold`，看低维样本是否更适合提前切到输入分裂。

预期收益：

1. 改进点足够具体，老师一看就知道不是“再跑一遍”。
2. 结果能直接体现在验证率、平均耗时、超时数和搜索节点数上。
3. 只需要改配置和实验组织，风险低，适合课程周期。

验收方式：

1. 至少 3 组分支配置对比。
2. 保持同一模型、同一测试集、同一 epsilon。
3. 汇总 VRA、timeout、mean/max time、visited nodes。

#### 方案 B：扰动半径自适应扫描

核心想法：把 epsilon 扫描做细，建立“扰动越大，鲁棒性如何衰减”的完整曲线，而不是只看单点结果。

可操作点：

1. 先固定样本集和策略，建议先用 kfsb。
2. 扫描 `epsilon = 0.005, 0.01, 0.02, 0.03, 0.05`。
3. 统计每个 epsilon 下的 verified acc、timeout、mean time 和 max time。

预期收益：

1. 更容易形成图表化结论，适合课堂展示。
2. 改进难度低，主要是实验设计和批量组织问题。
3. 能直接回答“模型在哪个半径开始明显失稳”。

验收方式：

1. 至少 5 个 epsilon 点。
2. 输出 VRA-epsilon 曲线和时间-epsilon 曲线。
3. 给出鲁棒性开始明显下降的拐点判断。

#### 方案 C：超时与预算分配优化

核心想法：聚焦难例处理，围绕 timeout、iteration 和 batch size 做预算分配实验，回答“如何在有限计算资源下多证明一点”。

可操作点：

1. 对比 `bab.timeout` 的不同取值，例如 15/30/60。
2. 调整 `solver.batch_size`、`alpha-crown.max_time`、`beta-crown.iteration`。
3. 观察 unknown/timeout 是否减少，以及是否值得为难例投入更多预算。

预期收益：

1. 能把“资源分配”这个工程问题讲清楚。
2. 结果不只看准确率，也看计算预算是否用得划算。
3. 适合把“改进”讲成一个明确的效率提升问题。

验收方式：

1. 至少 3 组预算配置。
2. 统计 verified acc、unknown/timeout、平均耗时。
3. 说明预算增加是否真的带来更多证明。

#### 选择建议

如果想让“改进”更像方法优化而不是纯参数复现，优先选方案 A。
如果想保证容易做完并且图表更直观，优先选方案 B。
如果想突出工程调度与资源控制能力，优先选方案 C。

我的建议是：**方案 A 作为主改进方向，方案 B 作为辅助结果线**。这样既有方法味道，也不会超出课程周期。
